SEE IT WORK

Paste a real error. Read the decision trace.

The Support Assistant normalizes your input, routes it through a decision trace, and returns root cause, fix, and prevention. The walkthrough below runs on a real ResultCode — try your own after.

Root cause

The customer cancelled or let the prompt expire.

ResultCode 1032 is the M-Pesa signal that a customer saw the STK prompt on their phone and did not enter a PIN — either they pressed cancel, closed the prompt, or let it time out. The transaction never left the rail as a debit attempt. The prompt did its job.

Fix
Surface the cancellation to the user with a clear retry affordance. Do not auto-retry — repeat prompts fatigue customers and lead to real cancellations on future sessions.
Prevention
Make the STK prompt copy specific (amount, merchant name, reference) so customers recognize the charge before the prompt arrives. Track 1032 rate by merchant; anything over 8% usually means prompt ambiguity, not customer intent.

ResultCode 1032 on STK Push

Customer cancelled or let the PIN prompt expire.

Read the fix
Root cause

The STK prompt reached the phone. The customer chose not to complete it. This is a UX signal, not a system failure.

Fix

Show the cancellation to the user; do not auto-retry.

Prevention

Write specific prompt copy (amount, merchant, reference). High 1032 rates usually mean prompt ambiguity.

Callback fired with ResultCode 0 — statement still pending

Callback-to-settlement race or B2C SMS delay.

Read the fix
Root cause

The callback is Daraja's acknowledgement, not the final settlement state. For B2C and C2B these can diverge by seconds in production, up to 30 seconds in sandbox.

Fix

Trust the callback. Don't block UI on the statement — it catches up.

Prevention

Display the TransactionID prominently so customers can verify on USSD `*334#` when the SMS lags.

Callback arrives but the body is malformed

STK, B2C, and Status callbacks have different shapes.

Read the fix
Root cause

STK Push callbacks use `stkCallback`, B2C callbacks use `Result`, and Status callbacks wrap the same `Result` shape under `stkCallbackResponse`. Parsing one as another drops fields silently.

Fix

Branch on the callback's top-level key before parsing. The SDK's `verifyCallback()` does this.

Prevention

Use the typed callback handlers in `@daraja-4/sdk` rather than hand-rolling JSON parsers per endpoint.

STK prompt reached the phone, PIN entered, still fails

Insufficient balance, daily limit, or wrong PIN.

Read the fix
Root cause

Five sub-causes: ResultCode 1 (balance), 8 (daily limit), 2001 (wrong PIN), 17 (system error), 20 (unable to lock). Each has a distinct customer-facing message.

Fix

Branch on the ResultCode; do not show a generic "failed" to the customer.

Prevention

Offer a fallback payment method (card, bank) for the cases the customer cannot recover from alone.

B2C reports success — recipient says nothing arrived

SMS notification lag or MSISDN formatting error.

Read the fix
Root cause

M-Pesa committed the funds to the wallet. The SMS that tells the recipient can lag by minutes during high load. Or the MSISDN was off by one digit and the money went to a valid but wrong account.

Fix

Ask the recipient to open M-Pesa app, not wait for SMS. Cross-check the MSISDN in your request against the `TransactionReceipt` in the callback.

Prevention

Validate MSISDN format before submission; display it back to the sender for confirmation.

SecurityCredential errors on B2C, balance, or reversal

Cert, initiator password, or sandbox/prod environment mismatch.

Read the fix
Root cause

The SecurityCredential is the initiator password encrypted with the sandbox or production public-key certificate. Wrong cert, wrong password, or a cert rotated between environments all produce the same error signature.

Fix

Use `daraja_explain("SecurityCredential")` for the generator walkthrough, or paste from the portal's Test Credentials page into `MPESA_SECURITY_CREDENTIAL`.

Prevention

Keep sandbox and production certificates in separately-named env vars; never reuse one for the other.

iOS eSIM and dual-SIM customers get 1037

STK routing picks the non-M-Pesa SIM on dual-active devices.

Read the fix
Root cause

STK Push routes to the MSISDN's home network. On iOS eSIM and dual-SIM Android, the Safaricom SIM can be the secondary; the prompt reaches the device but not the SIM M-Pesa expects.

Fix

For recurring payment customers, flag 1037 rates per MSISDN; those with repeated 1037s likely need a paybill fallback.

Prevention

Surface a one-line "if the prompt didn't arrive, open SIM 2" hint in your retry UI for high-1037 MSISDNs.

Sandbox passes, production fails the same call

Credentials, certificates, or amount limits differ between environments.

Read the fix
Root cause

Sandbox uses test credentials, a test cert, a test shortcode, and a test amount ceiling. Production requires real credentials issued post-go-live, the production cert, your own shortcode, and production limits apply.

Fix

Run `daraja_preflight` with both envs to diff what's different. Do not copy sandbox values into production.

Prevention

Use separate `.env.sandbox` and `.env.production` from day one. Tag your logs with the env.

ResultCode 0 but ResponseCode non-zero — or vice versa

ResponseCode is for the request; ResultCode is for the transaction.

Read the fix
Root cause

ResponseCode 0 means Daraja accepted your request. ResultCode 0 means the transaction itself succeeded. A request can be accepted (ResponseCode 0) and still fail (ResultCode 1032). They answer different questions.

Fix

Check both, in order: ResponseCode tells you whether you're entitled to wait for a callback; ResultCode tells you what the callback said.

Prevention

Name variables `requestAccepted` and `transactionSucceeded` rather than passing raw codes around.

Duplicate MerchantRequestID on retry

Retrying with the same UUID — or no UUID at all.

Read the fix
Root cause

Daraja deduplicates by MerchantRequestID for 30 seconds. Retries that reuse the ID are rejected with ResultCode 43 even though the original transaction may have failed.

Fix

Generate a fresh `crypto.randomUUID()` for every attempt. Log both old and new IDs against the same business-side reference.

Prevention

Use the SDK's built-in idempotency (automatic per-attempt UUIDs). Never hardcode a MerchantRequestID.

Callback vs polling race conditions

Status query fires before the callback has updated internal state.

Read the fix
Root cause

Callbacks are push; status queries are pull. Polling a transaction a second after initiation can return "pending" even when the callback is about to land.

Fix

Let the callback be the source of truth. Only poll as a reconciliation safety net, not as primary completion detection.

Prevention

If you poll, use exponential backoff starting at 5 seconds. The SDK's `collect()` already handles this for STK Push.

HTTP 401 or 403 on /oauth/v1/generate

Consumer key/secret mismatch, or passing them in the wrong header.

Read the fix
Root cause

OAuth expects the consumer key and secret in a Basic auth header, base64-encoded as `key:secret`. Common failure: copying the wrong app's credentials, or accidentally including a trailing newline in the env var.

Fix

Regenerate the app secret on developer.safaricom.co.ke; copy-paste into the env var with no surrounding whitespace.

Prevention

Use the SDK's `createClient({ consumerKey, consumerSecret })` rather than hand-rolling the Basic header. The SDK trims whitespace.

WHAT'S INSIDE

Five tools. One install.

Each tool is read-only or sandbox-only. None touch production. Diagnosis, explanation, and validation run entirely on bundled knowledge — no credentials required for the three tools developers reach for most.

Diagnose

daraja_diagnose

Paste an error code, a callback JSON body, or a log line. Returns root cause, fix, and prevention with a decision trace. Covers 27+ error codes including ambiguous ones (1032 vs 1037, ResultCode 1 on collect vs send).

Read-only Offline No credentials

Explain

daraja_explain

Plain-language walkthrough of any Daraja or M-Pesa concept — SecurityCredential, callbacks, OAuth, STK flow, going live. Grounded in the same concept docs Safaricom engineers use.

Read-only Offline No credentials

Validate

daraja_validate

Lint for M-Pesa integration code. Catches credential leaks, missing callback verification, amount-bound omissions, and URL issues. Partitions findings into blockers, warnings, and nits with a `canDeploy` signal.

Read-only Offline No credentials

Preflight

daraja_preflight

Pre-deployment health check. Validates code, pings the callback URL, and verifies OAuth against the sandbox — catches issues before they hit production. Sandbox-only.

Idempotent Sandbox-only

Test Sandbox

daraja_test_sandbox

Run a real sandbox transaction to verify a fix end-to-end. Uses the Safaricom test credentials. Marked destructive so your AI assistant asks before firing.

Destructive-hint Sandbox-only
SAFE BY DEFAULT

Read-only, offline-capable, zero credentials required.

The Support Assistant was designed so Safaricom can ship it to any developer, on any IDE, without worrying about credentials in AI contexts or data leaving the developer's machine. Five properties, each enforced at the package layer.

  • Read-only tool annotations The three most-used tools (diagnose, explain, validate) are marked readOnlyHint: true in MCP. AI assistants run them freely. The sandbox-touching tools (preflight, test_sandbox) are marked idempotentHint or destructiveHint so your AI assistant asks first.
  • PII sanitization by default Phone numbers are masked in every tool output before entering the AI context — 254708374149 becomes 254708***149. Tool outputs stream through the same sanitizer regardless of caller.
  • Decision traces as audit Every diagnosis returns its reasoning path: normalize_input → extract_code → error_db_hit → knowledge_base_lookup. The reasoning is inspectable by humans and by other AI tools.
  • Knowledge bundled at build time The 27+ error codes, 10 decision trees, and 11 symptom docs ship inside the npm package itself. The Support Assistant does not fetch over the network to answer a question.
  • No credentials, no data exfiltration Diagnose, explain, and validate operate on what you paste into them. Nothing is sent to Safaricom. Nothing is sent to a third-party LLM endpoint. Your secrets store is not read.

Full trust posture documented in knowledge/concepts/security.md.

Three lines. Every major IDE.

Pick your environment, drop the config, restart your AI assistant. Published under the @daraja-mcp/support@beta tag.

{ "mcpServers": { "daraja-support": { "command": "npx", "args": ["-y", "@daraja-mcp/support@beta"] } } }

Add to .mcp.json at your project root or ~/.claude/settings.json.

Add to ~/.codeium/windsurf/mcp_config.json.
